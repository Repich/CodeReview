# Стратегия тегирования системных стандартов

Источник: `docs/system_standards_index.md` содержит краткие резюме всех #std из
документа «Система стандартов…14_10_2025», а `norms.yaml` уже включает сгенерированные
блоки `STD_***` (см. пример `norms.yaml:826-910`). Для удобного поиска и привязки
к детекторам нужно добавить явные категории/теги и признак автоматизируемости.

## Новые поля
| Поле | Назначение |
| --- | --- |
| `category` | один из предопределённых доменов (см. ниже) |
| `tags` | список уточняющих меток (`ui`, `forms`, `process-only`, `performance`…) |
| `automation_hint` | `code`, `mixed`, `process` — насколько реалистично автоматизировать проверку |

`automation_hint` будет синхронизирован с `code_applicability`: для `process` норм
ставим `code_applicability: false`, иначе `true`.

## Категории / ключевые слова

| Категория | Ключевые слова / примеры |
| --- | --- |
| `governance_release` | релизы, версии, выпуск (`#std788`, `#std483`) |
| `metadata_design` | объекты метаданных, подсистемы, библиотеки (`#std470`, `#std543`) |
| `data_model_registers` | регистры, константы, предопределённые элементы (`#std603`, `#std632`) |
| `transactions_locks` | транзакции, блокировки (`#std783`, `#std460`) |
| `queries_performance` | запросы, индексы, виртуальные таблицы (`#std437`, `#std652`) |
| `background_jobs` | регламентные задания (`#std540`, `#std760`) |
| `security_access` | роли, права, RLS (`#std689`, `#std488`) |
| `integration_exchange` | обмен данными, EnterpriseData, классификаторы (`#std637`, `#std771`) |
| `localization` | локализация, языки, форматирование (`#std761`, `#std766`) |
| `ui_navigation` | командный интерфейс, панели, навигация (`#std711`, `#std715`) |
| `ui_forms_behavior` | формы, элементы, валидация, модальные окна (`#std630`, `#std703`) |
| `reporting_printing` | СКД, печатные формы, отчёты (`#std548`, `#std762`) |
| `performance_runtime` | длительные операции, клиент/сервер вызовы (`#std642`, `#std755`) |
| `devops_install` | установка, обновление, конфигурация окружения (`#std731`, `#std769`) |

Если стандарт не попадает в список, используем `governance_release` как резерв и
добавляем тег `needs-manual-review`.

## Алгоритм присвоения
1. Для каждого блока `STD_***` собрать текст (`title + norm_text`).
2. Сопоставить ключевые слова (регистронезависимые) — они задают:
   - первичную `category` (первая сработавшая группа);
   - теги из совпавших групп (`ui`, `queries`, `security` и т.д.).
3. Дополнительные теги по шаблонам:
   - наличие «Область применения» с одним приложением → `tags += ["scope:управляемое"]`.
   - слова `Методическая рекомендация`, `Рекомендуется` → добавить `advisory`.
4. `automation_hint = process`, если нет технических терминов (модули, код, формы, запросы).
   В остальных случаях `code`; промежуточные случаи (регламенты, локализация) помечаем `mixed`.
5. Обновить `norms.yaml` через ruamel.yaml, чтобы сохранить формат. Одновременно построить
   отчет `docs/norms_audit/tagging_summary.md` с количеством норм по категориям и
   списком `process` стандартов для последующего обсуждения.

После генерации ― показать отчёт и получить подтверждение перед переходом к маппингу норм ↔ детекторы.
