# Кандидаты правил (сформулировано по результатам LLM‑ревью)

Ниже список дополнительных норм, которые LLM регулярно упоминает в ревью 1С. Для каждой нормы указана область, краткое описание и пример нарушения.

## Транзакции и обработка ошибок

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-TXN-001 | Любая транзакция должна иметь явную пару `НачатьТранзакцию` / `ЗафиксироватьТранзакцию` / `ОтменитьТранзакцию`. В обработчике `Исключение` обязательно вызывать `ОтменитьТранзакцию()` | `Попытка НачатьТранзакцию(); ... Исключение Сообщить(ОписаниеОшибки()); КонецПопытки;` |
| LLM-TXN-002 | В ветке `Исключение` нужно как минимум записать техническую информацию в журнал регистрации и выдать пользователю «чистое» сообщение. Недостаточно только `СообщитьПользователю`. | `Исключение СообщитьПользователю("Ошибка"); КонецПопытки;` |

## Временные данные и дата/время

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-DATE-001 | В серверной бизнес-логике использовать `ТекущаяДатаСеанса()` вместо `ТекущаяДата()`, чтобы поддерживать таймзоны/режимы сервера. | `Документ.Дата = ТекущаяДата();` |

## Архитектура форм vs логика

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-FORM-001 | Модули форм не должны создавать/записывать НСИ и документы напрямую. Форма собирает данные и вызывает прикладной сервис. | В форме вызывают `Справочники.Номенклатура.СоздатьЭлемент().Записать()` |
| LLM-FORM-002 | Управление доступностью команд должно быть централизовано: одна процедура `УстановитьДоступностьЭлементов`, вместо рассыпанных `Элементы.*.Доступность = ...`. | Доступность кнопки меняется в нескольких обработчиках и не сбрасывается обратно в `Ложь`. |
| LLM-FORM-003 | Обязательные поля должны блокировать кнопку «Сохранить»: сначала установить `Доступность = Ложь`, затем включать при выполнении всех условий либо использовать `ОбщегоНазначенияКлиентСервер.УстановитьСвойствоЭлементаФормы`. | Процедура `УстановитьДоступностьСохранения` всегда ставит `Истина`. |

## Работа с данными форм и коллекциями

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-FORM-004 | Операции с `ДанныеФормыКоллекция` выполнять в один проход, не выгружать ТЗ несколько раз для одних и тех же расчётов. | Несколько циклов по одной и той же таблице значений для выставления флагов и подсчётов. |

## Комментарии, TODO и стиль кода

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-CODE-001 | В продуктивных модулях не должно быть закомментированных блоков рабочего кода. | `//Организация = СтруктураПараметров...` оставлено как комментарий. |
| LLM-CODE-002 | TODO/вопросы в коде допускаются только в трекере, но не в модуле. Любой `// TODO` должен быть закрыт. | `// Уточнить: менять ли количество?` |
| LLM-CODE-003 | Сложные тернарные выражения в одном операторе следует разносить на обычные `Если/Иначе`, чтобы повысить читаемость. | В одной строке вложенные `?(СтрНайти(...`, `?(...)`. |

## Запросы и производительность

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-QUERY-001 | Нельзя использовать `ВЫБРАТЬ *` в прикладных запросах — перечисляйте конкретные поля. | `Запрос.Текст = "ВЫБРАТЬ * ИЗ Справочник.Номенклатура";` |
| LLM-QUERY-002 | Не создавайте/не выполняйте запросы внутри циклов — подготовьте текст и параметры заранее. | В цикле `Для Каждого` на каждой итерации `Запрос = Новый Запрос; ... Запрос.Выполнить();`. |
| LLM-QUERY-003 | В отчетах/движениях использовать `УПОРЯДОЧИТЬ ПО`, если алгоритм зависит от порядка строк; и наоборот — не добавлять лишнее упорядочивание, если оно не нужно. | Отсутствует `УПОРЯДОЧИТЬ ПО`, хотя далее логика рассчитывает «первую» строку. |

## Регистры и НСИ

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-REG-001 | Набор записей регистра не создаётся и не записывается внутри циклов — используем один набор. (соответствует MULTI_WRITE_REG_01/03) | В `Для Каждого` создаётся `Регистры...СоздатьНаборЗаписей().Записать()`. |
| LLM-REG-002 | Имя реквизита/измерения не должно совпадать с именем владельца (чтобы избежать `Реквизиты.Номенклатура` в `Catalogs/Номенклатура`). | `Реквизиты.Номенклатура = ...` в модуле справочника Номенклатура. |

## Привилегированный режим и безопасность

| ID | Правило | Пример нарушения |
| --- | --- | --- |
| LLM-SEC-001 | Привилегированный режим включается точечно: `УстановитьПривилегированныйРежим(Истина)` → действие → `...Ложь`. Нельзя оборачивать им весь метод. | Привилегированный режим включён до выборки данных и выключается в конце процедуры. |
| LLM-SEC-002 | Любой серверный вызов/команда, выполняющая длительную операцию (>8 сек), должна идти через фоновое задание. | Формирование отчёта выполняется в модуле формы синхронно, без фонового задания. |

Этот список можно использовать как основу для расширения `norms.yaml` и бэклога детекторов. Каждый пункт уже встречается в ручных ревью и понятен разработчикам, что облегчает внедрение.
