# Процесс код-ревью (stat + LLM)

1. **Статический анализ (detectors)**
   - Запускаем воркер с текущим `norms.yaml` и набором эвристических детекторов.
   - Фиксируем все найденные нарушения в findings (JSONL).

2. **LLM-проверка пропущенных норм**
   - Для чувствительных сценариев (импорт данных, длительные операции, работа с файлами, безопасность) формируем вход:
     - исходный код/фрагменты,
     - выдержки из релевантных разделов `norms.yaml` (см. `docs/llm_long_operation_prompt.md` и аналогичные сводки),
     - список уже найденных детекторами нарушений.
   - Просим модель найти дополнительные нарушения и описать их в формате предложений новых норм (`norm_id`, `section`, `norm_text`, ссылка на стандарт).
   - Воркер сохраняет сырые ответы в таблицу `ai_findings` (статус `suggested`) и фиксирует `llm_prompt_version`.
   - Дополнительно каждый вызов LLM сохраняется в `artifact_storage/<run_id>_llm_*.json` + `IOLog`. Доступ к логам — только у администраторов.

3. **Фидбек от ревьюера**
   - В отчёт добавляем список найденных LLM норм и помечаем их как “требует подтверждения”.
   - Ревьюер отмечает:
     - ложные срабатывания текущих детекторов,
     - реальные нарушения, не пойманные автоматически.

4. **Интеграция новых норм**
   - Подтверждённые человеком нормы добавляем в `norms.yaml` вместе с ссылкой на источник и severity.
   - По возможности реализуем автоматический детектор; если пока невозможно, фиксируем `detector_type: manual_review`.

5. **Повторный прогон**
   - После пополнения `norms.yaml` повторяем стат-анализ на тех же исходниках, чтобы убедиться, что новые правила теперь покрываются на шаге 1.

6. **Выдача отчёта**
   - Итоговый отчёт включает:
     - автоматические findings,
     - подтверждённые LLM находки,
     - комментарии ревьюера.
   - Сводка действий по пополнению норм прилагается к PR/issue, чтобы аудит видел эволюцию правил.
   - В UI карточка запуска отображает предложения LLM отдельным блоком с кнопками подтверждения/отклонения; статус пишется обратно в `ai_findings`.
